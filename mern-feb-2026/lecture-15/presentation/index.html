Part 1 — Concept Challenge

Why do factory functions fall short?

What are the limitations of using factory functions for object creation?

  1. Factory functions can't accept parameters
  2. Factory functions are slower than object literals
  3. Duplicate methods per object, no instanceof, no inheritance ✓
  4. Factory functions can only create one type of object

1,000 objects = 1,000 copies of every method. No type identity. No hierarchy. Prototypes fix all three.

Part 1 — Quick-Fire Recall

What does this refer to inside greet()?


const user = { name: "Ali", greet() { return this.name; } };
user.greet();
  1. The global window object
  2. The user object (left of the dot) ✓
  3. The greet function itself
  4. undefined

Rule: this = whatever is LEFT of the dot when the function is called.

Part 1 Summary

  • Common Mistake: Forgetting that this depends on HOW the function is called, not WHERE it's defined
  • Best Practice: Use factory functions for simple data objects. When you need shared behavior or instanceof — use classes.
  • Pro Tip: The this binding bug is the #1 source of JavaScript confusion. Master it now — it saves hundreds of hours in React.

Part 2

Constructor Functions & Prototypes

How JavaScript shares behavior efficiently

📚 Analogy: A prototype is a shared library textbook. Every student has their own notebook (instance properties), but they share the textbook (prototype methods).

The Prototype Chain

👤 ali (instance) name, email, role — own properties
↓
🛠 User.prototype greet(), getInfo() — shared methods
↓
🌐 Object.prototype toString(), hasOwnProperty()
↓
🚫 null End of chain

Let's see this in code!

Constructor functions & the new keyword


function User(name, email, role) {
  this.name = name;
  this.email = email;
  this.role = role;
}
User.prototype.greet = function() {
  return `Hi, I'm ${this.name} (${this.role})`;
};

const ali = new User("Ali", "ali@x.com", "admin");
const sara = new User("Sara", "sara@x.com", "dev");
ali.greet === sara.greet; // true — shared!

What new Actually Does

📦 1. Create {} Empty object
→
🔗 2. Link Prototype Wire to .prototype
→
🔨 3. Run Constructor this = new obj
→
✅ 4. Return Object Automatically
⚠ Forget new with a constructor function? this points to the global object. ES6 classes fix this — they throw a TypeError.

Part 2 — Predict Output

True or false: user1.greet === user2.greet?


function User(name) { this.name = name; }
User.prototype.greet = function() { return `Hi, ${this.name}`; };
const user1 = new User("Ali");
const user2 = new User("Sara");
  1. TRUE — both share the same function on User.prototype ✓
  2. FALSE — each instance gets its own copy
  3. Error — you can't compare functions
  4. undefined — greet doesn't exist on instances

Prototype methods are shared. 10,000 users = ONE greet function in memory!

Part 2 — Spot the Error

What happens when you forget new?


function User(name) { this.name = name; }
const ali = User("Ali"); // No new!
console.log(ali);
  1. { name: "Ali" }
  2. undefined — function returns nothing, name leaks to global ✓
  3. TypeError: User is not a constructor
  4. null

Without new, this becomes global. The function returns nothing. PascalCase naming prevents this confusion.

Part 2 Summary

  • Common Mistake: Defining methods inside the constructor (this.greet = function...) — wastes memory, every instance gets its own copy
  • Best Practice: Always put shared methods on .prototype. Use Object.getPrototypeOf() instead of __proto__.
  • Pro Tip: PascalCase for constructors (User, not user). Never modify Object.prototype — it breaks everything.

Class Syntax


class User {
  constructor(name, email, role) {
    this.name = name;       // instance property
    this.email = email;
    this.role = role;
  }
  greet() {                 // prototype method (shared)
    return `Hi, I'm ${this.name} (${this.role})`;
  }
  static validateEmail(email) {  // class-level utility
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
  }
}

User.validateEmail("ali@x.com"); // true (static)
new User("Ali", "ali@x.com", "admin").greet(); // instance

Role Hierarchy — extends & super


class Admin extends User {
  constructor(name, email) {
    super(name, email, "admin"); // MUST be first!
    this.permissions = ["create","read","update","delete"];
  }
  deleteUser(user) {
    return `${this.name} deleted ${user.name}`;
  }
  greet() { // Override — polymorphism!
    return `Admin ${this.name} — Full access`;
  }
}

class Developer extends User {
  constructor(name, email, skills = []) {
    super(name, email, "developer");
    this.skills = skills;
  }
  greet() {
    return `Dev ${this.name} — ${this.skills.join(", ")}`;
  }
}

Polymorphism in Action


const team = [
  new Admin("Ali", "ali@x.com"),
  new Manager("Sara", "sara@x.com", "Engineering"),
  new Developer("Omar", "omar@x.com", ["JS", "React"])
];

// Same method name, different output!
team.forEach(m => console.log(m.greet()));
// "Admin Ali — Full access"
// "Manager Sara — Engineering Department"
// "Dev Omar — JS, React"
💎 instanceof chain: admin instanceof Admin ✓ & admin instanceof User ✓ — checks the ENTIRE prototype chain

Calling Parent Methods with super


class TeamLead extends Developer {
  constructor(name, email, skills, teamSize) {
    super(name, email, skills);
    this.teamSize = teamSize;
  }
  greet() {
    const devGreet = super.greet(); // Parent's greet
    return `${devGreet} | Leading ${this.teamSize}`;
  }
}

const lead = new TeamLead("Zain", "z@x.com", ["React"], 5);
lead.greet();
// "Dev Zain — React | Leading 5"

lead instanceof TeamLead;  // true
lead instanceof Developer; // true
lead instanceof User;      // true — full chain!

Part 3 — Spot the Error

What's wrong with this class?


class Admin extends User {
  constructor(name, email) {
    this.name = name;     // Using this before super()
    this.permissions = ["all"];
  }
}
  1. Nothing — this code works fine
  2. ReferenceError — must call super() before using this ✓
  3. TypeError — Admin can't extend User
  4. SyntaxError — constructor can only have one argument

Hard rule: super() MUST come before any this usage in subclass constructors.

Part 3 — Concept Challenge

Why is validateEmail a static method?

  1. Because it's a private method
  2. Because it can only be called once
  3. Because it doesn't need an instance — no this required ✓
  4. Because it returns a boolean

Static = class-level utility. Ask: "Does this method need this?" If no, make it static.

Part 3 Summary

  • Common Mistake: Forgetting super() in a subclass — causes ReferenceError. Or using this before calling it.
  • Best Practice: One class per responsibility. Use super.method() to extend parent behavior rather than rewriting.
  • Pro Tip: This User → Admin/Manager/Developer hierarchy is the exact RBAC (Role-Based Access Control) pattern in enterprise apps.

Part 4

Encapsulation, Composition & Observer

Private fields, building with pieces, and reactive updates

Public (Default)Private (# prefix)
this.balance = 1000 this.#balance = 1000
Anyone can modify Only the class can access

Private Fields & Validated Setters


class Task {
  #id;
  #status;

  constructor(title) {
    this.#id = crypto.randomUUID();
    this.title = title;         // public
    this.#status = "pending";   // private
  }

  get id() { return this.#id; }
  get status() { return this.#status; }

  set status(val) {
    const valid = ["pending","in-progress","completed"];
    if (!valid.includes(val)) throw new Error("Invalid");
    this.#status = val;
  }
}

const task = new Task("Build Login");
task.status = "in-progress"; // ✓ setter validates
// task.#id;  // SyntaxError — truly private!

Composition vs Inheritance

Inheritance (IS-A)


class Admin extends User { }
// Admin IS-A User
// Single parent only
// Tight coupling
VS

Composition (HAS-A)


class TaskManager {
  #logger = new Logger();
  #validator = new Validator();
}
// TaskManager HAS-A Logger
// Combine unlimited pieces
// Loose coupling
💡 When in doubt, prefer composition. JavaScript only allows single inheritance, but you can compose unlimited pieces.

Observer Pattern — The React Connection


class EventEmitter {
  #listeners = {};

  on(event, callback) {
    if (!this.#listeners[event]) this.#listeners[event] = [];
    this.#listeners[event].push(callback);
    return () => { // Unsubscribe function!
      this.#listeners[event] =
        this.#listeners[event].filter(cb => cb !== callback);
    };
  }

  emit(event, data) {
    this.#listeners[event]?.forEach(cb => cb(data));
  }
}

Observer → React Mapping

Observer PatternReact Equivalent
emitter.on(event, fn) useEffect(() => { subscribe() })
Returned unsubscribe fn useEffect cleanup: return () => unsub()
emitter.emit(event, data) setState(newData)
Listeners re-fire on emit Components re-render on state change
💎 This pattern powers React, Redux, RxJS, and every event-driven system. You're learning the architecture of the modern web.

Part 4 — Concept Challenge

What's the difference between this.balance and this.#balance?

  1. # makes it read-only
  2. # is just a naming convention, no enforcement
  3. # is truly private — SyntaxError from outside the class ✓
  4. # makes the field a constant

Private fields + validated setters = bulletproof objects. Control ALL access through methods.

Part 4 — Predict Output

How many times does "Change!" print?


const emitter = new EventEmitter();
const unsub = emitter.on("change", () => console.log("Change!"));
emitter.emit("change");
emitter.emit("change");
unsub();
emitter.emit("change");
  1. 1 time
  2. 2 times — unsubscribe stops the third emit ✓
  3. 3 times
  4. 0 times — emit doesn't trigger listeners

This is exactly React's useEffect pattern: subscribe on mount, unsubscribe on cleanup.

Part 4 — Concept Challenge

TaskManager needs logging, validation & notifications. Inherit or compose?

  1. Inherit from Logger — extends Logger
  2. Inherit from all three — extends Logger, Validator, Notifier
  3. Compose — use Logger, Validator, Notifier as internal components ✓
  4. Don't use any — put all logic directly in TaskManager

IS-A vs HAS-A: A TaskManager IS-NOT a Logger — it HAS-A Logger. JS only allows single inheritance anyway.

Part 4 Summary

  • Common Mistake: Returning private arrays directly from getters — always return copies: return [...this.#tasks]
  • Best Practice: Always unsubscribe from observers to prevent memory leaks — same as React's useEffect cleanup.
  • Pro Tip: The Observer pattern is THE most important design pattern for frontend developers. It powers every UI framework.
