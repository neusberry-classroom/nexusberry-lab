Part 1 — Spot the Error

This code crashes. Why?


const user = { name: "Zara", settings: undefined };
const { settings: { theme } } = user;
  1. theme is undefined (no crash)
  2. TypeError — can't destructure undefined ✔
  3. settings becomes null
  4. SyntaxError — invalid destructuring

Fix: const { settings: { theme } = {} } = user; — provide a default empty object.

Part 1 Summary

⚠️ Common Mistake: After renaming { role: jobTitle }, only jobTitle exists — role is not a variable.
✅ Best Practice: Always provide defaults when destructuring API responses. Defensive destructuring prevents production crashes.
💎 Pro Insight: The #1 source of "undefined is not an object" errors in production is destructuring API responses without defaults.

Part 2

Spread, Rest & The Art of Copying

Three dots, two superpowers

When you assign an object to a new variable, you get a reference, not a copy.
Change one — the other changes too. The spread operator fixes this.

Spread vs Rest — Same Syntax, Different Context

ContextSyntaxNameWhat It Does
Array / object literal...arrSpreadExpands elements
Function parameter...argsRestCollects into array
Destructuring...restRestCollects remaining
💡 Building/creating something? Spread. Receiving/collecting? Rest.

Shallow vs Deep Copy

Shallow (Spread)

Top level: independent ✅

Nested: still shared 💥

{ ...obj }

vs

Deep (structuredClone)

Top level: independent ✅

Nested: also independent ✅

structuredClone(obj)

⚠️ Spread copies one level deep. Pushing to a nested array in a spread copy mutates the original!

Part 2 — Interactive Question

What does original.tags contain?


const original = { name: "Report", tags: ["urgent", "Q1"] };
const copy = { ...original };
copy.tags.push("reviewed");
copy.name = "Report v2";
  1. ["urgent", "Q1"] — copy is independent
  2. ["urgent", "Q1", "reviewed"] — nested array is shared ✔
  3. ["reviewed"] — tags was replaced
  4. TypeError — frozen object

original.name is safe (top-level string), but original.tags is mutated — shallow copy trap!

Part 2 — Concept Challenge

Remove password, change role to "admin"


const user = { name: "Ali", email: "ali@test.com",
               password: "secret", role: "user" };
  1. delete user.password; user.role = "admin";
  2. const admin = { ...user, password: null, role: "admin" };
  3. const { password, ...safe } = user; const admin = { ...safe, role: "admin" }; ✔
  4. Object.assign(user, { role: "admin" });

Rest strips the field, spread overrides the role — no mutation, password is truly gone.

Part 2 Summary

⚠️ Common Mistake: Spread is shallow — nested objects/arrays are still shared references.
✅ Best Practice: Always return new objects/arrays instead of mutating. { ...defaults, ...overrides } — put defaults first.
💎 Pro Insight: React won't re-render if you mutate state directly. Spread + destructuring patterns are the foundation of every React state update.

Part 3 — Interactive Question

What does result look like?


const prices = { apple: 2, banana: 1, cherry: 4, date: 3 };
const result = Object.fromEntries(
    Object.entries(prices)
        .filter(([fruit, price]) => price > 2)
        .map(([fruit, price]) => [fruit, price * 100])
);
  1. { apple: 200, cherry: 400, date: 300 }
  2. { cherry: 4, date: 3 }
  3. { cherry: 400, date: 300 } ✔
  4. [["cherry", 400], ["date", 300]]

Filter keeps cherry (4) & date (3), map multiplies by 100, fromEntries converts back to object.

Part 3 — Spot the Error

This should NOT mutate target. What's wrong?


const target = { a: 1 };
const source = { b: 2 };
const merged = Object.assign(target, source);
  1. Nothing wrong — target is untouched
  2. Object.assign mutates target — both target and merged are { a:1, b:2 } ✔
  3. source gets modified instead
  4. TypeError — assign is not a function

Fix: Object.assign({}, target, source) or better: { ...target, ...source }

Part 3 Summary

⚠️ Common Mistake: Object.freeze is shallow — nested arrays/objects can still be mutated.
✅ Best Practice: Use Object.entries + HOFs over for...in — safer (no inherited properties) and chainable.
💎 Pro Insight: Object.freeze on config objects prevents one of the hardest bugs to trace: silent config mutation across modules.

Part 4 — Interactive Question

What does JSON.parse(JSON.stringify(data)) return?


const data = {
    name: "Test", count: 0, active: false,
    items: undefined, greet: () => "hello"
};
  1. Same as data — all 5 properties
  2. { name: "Test" } — only strings survive
  3. { name: "Test", count: 0, active: false } — undefined & functions dropped ✔
  4. TypeError — can't stringify functions

0 and false are valid JSON. undefined and functions are silently dropped.

Part 4 Summary

⚠️ Common Mistake: Shallow merging nested configs wipes out deep defaults. Spread at each nested level.
✅ Best Practice: Use structuredClone for deep cloning. JSON.parse(JSON.stringify()) drops functions, Dates, and undefined.
💎 Pro Insight: Mastering destructuring + spread + Object.entries + fromEntries handles 95% of data transformations — no libraries needed.

Part 5 — Spot the Error

Which line will TypeScript reject?


interface Product {
    readonly id: string;
    name: string;
    price?: number;
}
const item: Product = { id: "P001", name: "Widget" };
item.name = "Super Widget";  // Line A
item.price = 29.99;          // Line B
item.id = "P002";            // Line C
  1. Line A — can't modify after creation
  2. Line B — optional means read-only
  3. Line C — id is readonly, can't reassign ✔
  4. All three lines are fine

name is mutable, price is optional (can be added). Only readonly id can't be reassigned.

Part 5 Summary

⚠️ Common Mistake: readonly is compile-time only — it doesn't add runtime protection like Object.freeze.
✅ Best Practice: Use interface for object shapes, type for unions and intersections. Mark IDs as readonly.
💎 Pro Insight: Well-defined interfaces are the difference between a maintainable codebase and chaos. They're documentation the compiler enforces.
