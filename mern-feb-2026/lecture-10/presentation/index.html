Part 1

Why Programs Need Repetition

The for Loop

Your Admission Gateway processed one student.
A real university processes thousands. That's what loops are for.

💡 Analogy: A factory assembly line — each station performs the same task on every car that comes through. Same code, different data, repeated until done.

JavaScript's 5 Loop Types

LoopUse Case
forKnown number of iterations (counting)
whileUnknown iterations, condition-first
do...whileUnknown iterations, body-first (runs at least once)
for...inIterate over object keys
for...ofIterate over array/string values

Anatomy of a for Loop


for (let i = 0;  i < 5;  i++)
Initialize (runs once)
Test (before each run)
Update (after each run)
⚠️ Condition is checked BEFORE the body runs. A for loop can run zero times if the condition starts false.

Loop Execution Flow

🔄 Initialize let i = 0
→
❓ Condition? i < 5
→
✅ Run Body { ... }
→
🔼 Update i++

After Update, loop jumps back to Condition. When condition is false → EXIT.

🤔 Predict the Output


for (let i = 3; i >= 0; i--) {
    console.log(i);
}
  • 3, 2, 1
  • 3, 2, 1, 0
  • 3, 2, 1, 0, -1
  • Infinite loop
💡 0 >= 0 is true, so 0 gets printed. Then i-- makes it -1, and -1 >= 0 is false — NOW the loop stops.

🔎 Spot the Error

This should print 1 through 5. What's wrong?


for (let i = 1; i <= 5; i--) {
    console.log(i);
}
  • Missing break
  • Should use < not <=
  • i-- should be i++ — infinite loop!
  • Start at 0 not 1
💥 Wrong update direction = infinite loop. Always verify: does your update move toward the condition becoming false?

Part 1 Summary

⚠️ Common Mistake: Using <= with array.length — gives undefined on last iteration. Always use <.
✅ Best Practice: Use const inside loop bodies. The loop variable is let, but derived values should be const.
💎 Pro Tip: Name nested loop variables meaningfully: row, col instead of i, j.

Part 2

while, do...while & Loop Control

What if you don't know how many times to loop?
Search until found. Validate until valid. Retry until success.

💡 Analogy: while = check the weather before going outside. do...while = go outside first, then check.

while vs do...while

while

Condition checked before body

May run 0 times

"Check first, then act"

VS

do...while

Condition checked after body

Always runs at least 1 time

"Act first, then check"

break & continue

⏹️ break Exit the entire loop immediately. No more iterations.
⏩ continue Skip the rest of this iteration. Move to the next one. 

// break — stop searching once found
for (let i = 0; i < stockLevels.length; i++) {
    if (stockLevels[i] === 0) {
        console.log(`Out of stock at index ${i}`);
        break;  // Exit loop
    }
}

// continue — skip out-of-stock items
for (let i = 0; i < inventory.length; i++) {
    if (inventory[i].stock === 0) continue;  // Skip this one
    console.log(inventory[i].name);
}

Nested Loops

Outer loop × inner loop = total iterations


// Warehouse Grid: 3 rows × 4 columns = 12 iterations
for (let row = 1; row <= 3; row++) {
    let rowDisplay = `Row ${row}: `;
    for (let col = 1; col <= 4; col++) {
        rowDisplay += `[${row}-${col}] `;
    }
    console.log(rowDisplay);
}
// Row 1: [1-1] [1-2] [1-3] [1-4]
// Row 2: [2-1] [2-2] [2-3] [2-4]
// Row 3: [3-1] [3-2] [3-3] [3-4]
⚠️ 1,000 × 1,000 = 1 million iterations. Be careful with nested loops on large datasets.

⚡ Quick-Fire: break vs continue

What does continue do inside a loop?

  • Exits the loop entirely
  • Restarts the loop from the beginning
  • Skips the rest of the current iteration, moves to next
  • Pauses the loop for 1 second
💡 break = stop the whole loop. continue = skip this one item, keep going.

Part 2 Summary

⚠️ Common Mistake: Forgetting to update the condition variable in while loops — guaranteed infinite loop.
✅ Best Practice: Default to for for known counts. Use while for unknown-length. do...while only when body must run at least once.
💎 Pro Tip: while(true) with an internal break is a valid pattern for complex exit conditions.

Part 3

Arrays — Ordered Collections

One variable = one value. But an inventory has hundreds of products.
Arrays store multiple values in order, accessed by position.

💡 Analogy: A numbered filing cabinet — Drawer 0, Drawer 1, Drawer 2. The number (index) tells you exactly where to find each item.

Array Index Mapping

Index 0 Laptop
Index 1 Mouse
Index 2 Keyboard
Index 3 Monitor
Index 4 Webcam

products.length = 5  |  products[0] = "Laptop"  |  products[4] = "Webcam"  |  products[5] = undefined

const items = ["A", "B", "C", "D"];
items.splice(1, 2);

🧠 Concept Challenge

Why does products[products.length] return undefined?

  • It throws a RangeError
  • JS doesn't throw errors for out-of-bounds — returns undefined silently
  • It returns null
  • It returns the last element
⚠️ Silent failures are worse than loud errors. Use < not <= with length to avoid this trap.

Part 3 Summary

⚠️ Common Mistake: Confusing splice (mutates) with slice (non-mutating) — one letter difference!
✅ Best Practice: push/pop are O(1). unshift/shift are O(n). Prefer end operations.
💎 Pro Tip: In React, mutating arrays breaks re-rendering. Non-mutating methods are the foundation of state updates.

🧠 Concept Challenge

In the Category Summary, why check categories[cat] === undefined before adding?

  • To prevent duplicate categories
  • First encounter would crash: undefined.count is a TypeError
  • JavaScript requires it for object property access
  • It's not needed — optional optimization

🤔 After Shipment Processing

Monitor started at stock: 0. Shipment adds 20. What's the final stock?

  • 0 — shipment creates a new product
  • 20 — shipment updates the existing product in place
  • undefined — out-of-stock items can't be updated
  • Error — can't add to a zero value
💡 Array mutation in action — we changed the original object. In React, we'd create a new array instead.

✅ Best Practice: Match loop to task — for for indexed lists, while for search, for...of for clean iteration, for...in for object keys.
💎 Pro Tip: Extract repeated access: const product = inventory[i] at the top of the loop body.

Part 5

for...in, for...of & TypeScript Arrays

Two loops that look identical — one word different — completely different behavior.

for...in vs for...of

for...in

Iterates keys

On arrays: string indices "0", "1"

On objects: property names

Best for: Objects

VS

for...of

Iterates values

On arrays: actual elements

On objects: TypeError!

Best for: Arrays, strings

Remember: in = keys in an object. of = values of a collection.

Side-by-Side Comparison


const fruits = ["Apple", "Banana", "Cherry"];

// for...in — gives KEYS (string indices for arrays)
for (const key in fruits) {
    console.log(key);       // "0", "1", "2"
    console.log(typeof key); // "string" (not number!)
}

// for...of — gives VALUES (the actual elements)
for (const value of fruits) {
    console.log(value);  // "Apple", "Banana", "Cherry"
}
⚠️ for...in on arrays gives string indices, not numbers. Use for...of for arrays.

TypeScript Array Types (Preview)


// Typed arrays — two equivalent syntaxes
const prices: number[] = [89999, 1999, 24999];
const names: Array<string> = ["Laptop", "Mouse"];

// Tuples — fixed-length, typed at each position
const product: [string, number] = ["Laptop", 89999];

// readonly — prevent mutation
const CATEGORIES: readonly string[] = ["Electronics", "Furniture"];
// CATEGORIES.push("Food"); ← TypeScript ERROR!
💎 readonly prevents all mutation methods (push, pop, splice). Standard practice in React codebases.

💡 Hidden Fact Reveal

What happens with for...of on a plain object?


const obj = { a: 1, b: 2 };
for (const val of obj) { console.log(val); }
  • Logs 1, 2
  • Logs "a", "b"
  • TypeError: obj is not iterable
  • Logs undefined
💥 Plain objects are NOT iterable! Use for...in for objects, for...of for arrays. They are not interchangeable.

Part 5 Summary

⚠️ Common Mistake: Using for...in on arrays — gives string indices, not values. Using for...of on objects — throws TypeError.
✅ Best Practice: for...of for arrays and strings. for...in for objects. Never mix them.
💎 Pro Tip: Use number[] syntax over Array<number> for simple types. Use readonly for arrays that should never change.
