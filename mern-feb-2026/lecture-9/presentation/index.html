What's the logic bug?


const age = 25;
const hasID = true;

if (age >= 18) {
    console.log("Age verified");
}
if (hasID) {
    console.log("ID verified");
}
console.log("Access granted");
  1. It crashes on line 3
  2. "Access granted" always prints — regardless of conditions ✔
  3. hasID should use ===
  4. Missing semicolons

Two separate if blocks are NOT a connected chain. Use if/else or &&.

Part 1 Summary

  • Common Mistake: Checking lowest threshold first — catches everything
  • Common Mistake: Using = (assignment) instead of === (comparison)
  • Best Practice: Always use curly braces, even for single-line blocks
  • Best Practice: Nest max 2 levels — flatten with &&
💎 25 years insight: The most common logic bugs aren't inside the if — they're in the condition itself. Ask: what are ALL possible values?

Part 2

switch Statements & Ternary

When you're comparing one value against a list of known options,
switch is the right tool. For simple two-way values: ternary.

switch vs if/else

Use if/else

Range checks (>, <, ≥)

Complex boolean expressions

Different variables per condition

2–3 branches

VS

Use switch

Exact value matching

One variable, many values

Comparing against constants

4+ branches

Anatomy of a switch


switch (department) {
    case "Computer Science":
        console.log("CS — Building 4");
        break;
    default:
        console.log("Unknown department");
}
Keywords (switch, break) Expression (strict ===) case / default Match value
⚠ Without break, code "falls through" to the next case. Usually a bug!

Part 2 — Interactive Question

What gets logged?


const color = "green";
switch (color) {
    case "red":
        console.log("Stop"); break;
    case "yellow":
        console.log("Caution");
    case "green":
        console.log("Go");
    case "blue":
        console.log("Rare signal"); break;
    default:
        console.log("Unknown");
}
  1. Only "Go"
  2. "Go" then "Rare signal" ✔
  3. "Go", "Rare signal", "Unknown"
  4. "Caution" then "Go"

No break after "Go" → falls through to "blue" case, then break stops it.

Part 2 — Concept Challenge

Rewrite as ternary:


let message;
if (score >= 50) {
    message = "Pass";
} else {
    message = "Fail";
}
  1. let message = score >= 50 ? "Pass" : "Fail";
  2. const message = score >= 50 ? "Pass" : "Fail"; ✔
  3. const message = score >= 50 ?? "Pass" : "Fail";
  4. const message = score >= 50 || "Pass";

Bonus: Ternary lets us use const — one expression, one assignment!

Part 2 Summary

  • Common Mistake: Forgetting break — causes fall-through bugs
  • Nested Ternary: OK for 2–3 branch value assignment (vertical format). Avoid 4+ or side effects.
  • Best Practice: Always include default in every switch
  • Best Practice: Ternary for values only — not for side effects
💎 25 years insight: Every switch must have default, every case must have break (or a // falls through comment). That simple team rule eliminates an entire class of bugs.

Part 3

Short-Circuit, ?? & ?.

&& and || don't just return true/false — they return values.
Combined with ?? and ?., they're the modern JS decision toolkit.

Short-Circuit Rules

|| OR — First Truthy If A is truthy, return A.
Otherwise, return B.
&& AND — First Falsy If A is falsy, return A.
Otherwise, return B.
ExpressionResultWhy
null || "default""default"null is falsy
"Rana" && 4242Both truthy, returns last
0 || 42420 is falsy

|| vs ?? — The Critical Difference

|| (OR)

Triggers on all falsy values:

0, "", false, null, undefined, NaN


const seats = 0;
seats || "No data";
// "No data" ❌ (0 is valid!)
VS

?? (Nullish)

Triggers only on:

null, undefined


const seats = 0;
seats ?? "No data";
// 0 ✅ (preserved!)

Optional Chaining ?.

Stops evaluation & returns undefined when it hits null/undefined.


// Property access
student.address?.city           // "Lahore" or undefined

// Array element
student.scores?.[0]             // 85 or undefined

// Method call
student.getGPA?.()              // 3.5 or undefined

// The power combo: ?. + ??
const city = student.address?.city ?? "City not provided";
⚡ Pro Tip: ?. + ?? is the most common pattern in modern JS. Use it for all external/optional data.

Part 3 — Interactive Question

What's the output of each line?


console.log(0 || 42);
console.log(0 ?? 42);
  1. Both log 42
  2. Both log 0
  3. 42 then 0 ✔
  4. 0 then 42

|| treats 0 as falsy → returns 42. ?? only checks null/undefined → keeps 0.

Part 3 — Hidden Fact Reveal

What does this return?


console.log("" ?? "fallback");
  1. "fallback"
  2. undefined
  3. "" (empty string) ✔
  4. null

?? preserves empty strings and 0 — they're not null/undefined!

Part 3 Summary

  • Common Mistake: Using || when ?? is needed — loses valid 0 and ""
  • Best Practice: Default to ?? — it's safer than ||
  • Best Practice: ?. + ?? for all external/optional data
  • Logical Assignment: ??= assigns only if null/undefined
💎 25 years insight: A scholarship system used || instead of ??. Students with 0% scholarship got 100%. One character difference — millions in lost tuition.

Part 5 — Preview

TypeScript Narrowing

TypeScript can refine types based on your conditional checks.
After if (typeof x === "string"), TS knows x is a string inside that block.

3 Narrowing Techniques

🔎 typeof Guards Check primitive types
typeof x === "string"
✓ Truthiness Eliminate null/undefined
if (address) { ... }
📈 Discriminated Unions Object variants with switch
result.status === "admitted"

Discriminated Unions + switch


type AdmissionResult =
    | { status: "admitted"; department: string; scholarship: number }
    | { status: "waitlisted"; position: number }
    | { status: "rejected"; reason: string };

function processResult(result: AdmissionResult): string {
    switch (result.status) {
        case "admitted":
            return `Welcome to ${result.department}!`;  // TS knows shape
        case "waitlisted":
            return `Position #${result.position}`;       // TS knows shape
        case "rejected":
            return `Rejected: ${result.reason}`;         // TS knows shape
    }
}
⚡ Exhaustiveness: Remove a case → TypeScript shows a compile-time error. Free safety net!

Part 5 — Concept Challenge

What happens if we remove case "rejected"?

  1. Nothing — default handles it
  2. Runtime crash when status is "rejected"
  3. TypeScript compile-time error — unhandled case ✔
  4. It silently returns undefined

TypeScript checks that every variant is handled. Add a new status? Every switch will error until updated.

Part 5 Summary

  • typeof guards → primitive type checks
  • Truthiness → null/undefined elimination
  • Discriminated unions + switch → object variant handling
💎 React preview: { status: 'loading' } | { status: 'success', data } | { status: 'error', message } — this exact pattern models component states in Module 3.
