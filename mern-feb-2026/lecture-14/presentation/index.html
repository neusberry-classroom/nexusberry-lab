Why Async Matters

Single-Threaded JS & The Event Loop

Without Async (Blocking)With Async (Non-Blocking)
Request weather → page freezes → response Request weather → page stays interactive → update
White screen, spinning cursor Loading spinner, responsive UI
🍳 Analogy: JavaScript is a restaurant with ONE chef. Async = put cake in oven, start next dish. The oven is a Web API, the timer ringing is the callback queue.

The JavaScript Runtime

📋 Call Stack Executes code
→
🌐 Web APIs Timer, fetch, DOM
→
📦 Callback Queue Waiting callbacks
→
🔄 Event Loop Stack empty? Move next
⚠ setTimeout(fn, 0) does NOT run immediately — it still goes through Web API → Queue → Event Loop. Synchronous code always runs first!

Synchronous vs Asynchronous

Synchronous (Blocking)


Task A ████████
Task B         ████████
Task C                 ████
Total: 12 seconds
VS

Asynchronous (Non-Blocking)


Task A ████
Task B     ████ (Web API)
Task C ████████
Total: ~8 seconds

Part 1 — Concept Challenge

Why doesn't setTimeout(fn, 0) execute immediately?

  1. JavaScript is too slow to process it
  2. The browser adds a minimum 4ms delay
  3. The callback must go through Web API → Queue → Event Loop ✓
  4. setTimeout is broken for 0ms values

The event loop only moves callbacks to the stack when the stack is empty. It's about the path, not the delay.

Part 2

Callbacks & The Evolution to Promises

From the Pyramid of Doom to flat, readable chains

Callbacks The original pattern
Promises (ES2015) Flat chaining
async/await (ES2017) Reads like sync

Callback Hell vs Promise Chain

Callbacks (Nested)


getUser(id, (user) => {
  getOrders(user.id, (orders) => {
    getDetails(orders[0].id,
      (details) => {
        console.log(details);
    });
  });
});
VS

Promises (Chained)


getUser(id)
  .then(user =>
    getOrders(user.id))
  .then(orders =>
    getDetails(orders[0].id))
  .then(details =>
    console.log(details))
  .catch(err =>
    console.error(err));

Promise States

⏳ PENDING Operation in progress (order placed)
↓
✅ FULFILLED → .then() Completed successfully (delivered!)
↓
❌ REJECTED → .catch() Failed (canceled, error)

.finally() runs in BOTH cases — cleanup regardless of outcome

Let's see this in code!

Callbacks → Promises → Chaining


// Creating a Promise
function getWeather(city) {
  return new Promise((resolve, reject) => {
    if (city === "") reject(new Error("City required"));
    else resolve({ city, temp: 32 });
  });
}

// Consuming with .then/.catch/.finally
getWeather("Lahore")
  .then(data => console.log(data.temp))
  .catch(err => console.error(err.message))
  .finally(() => console.log("Done"));

Part 2 — Spot the Error

What's wrong with this Promise?


function loadData(url) {
  return new Promise((resolve) => {
    fetch(url)
      .then(response => response.json())
      .then(data => resolve(data));
  });
}
  1. fetch() is not allowed inside a Promise
  2. No reject path — if fetch fails, the Promise hangs forever ✓
  3. resolve(data) should be return data
  4. Missing async keyword

Part 2 — Predict Output

What's the output?


Promise.resolve("Hello")
  .then(v => { console.log("A:", v); throw new Error("Oops"); })
  .then(v => { console.log("B:", v); })
  .catch(e => { console.log("C:", e.message); })
  .finally(() => { console.log("D: Done"); });
  1. A: Hello, B: undefined, D: Done
  2. A: Hello, C: Oops, D: Done ✓
  3. A: Hello, C: Oops
  4. C: Oops, D: Done

throw converts to rejection → B skipped → catch handles → finally always runs

Part 2 — Concept Challenge

Why does .then() return a NEW Promise?

  1. It's a JavaScript engine bug that was kept for compatibility
  2. To prevent memory leaks from circular references
  3. Promises are immutable once settled — new Promise enables chaining & branching ✓
  4. So the garbage collector can clean up the old one

Part 2 Summary

  • Common Mistake: Forgetting to return inside .then() — next handler gets undefined
  • Best Practice: Every Promise chain must end with .catch() — unhandled rejections are silent killers
  • Pro Tip: Use arrow functions with implicit return for clean chains: .then(data => data.json())

Part 3

Promise Combinators & async/await

Parallel execution and modern syntax

The Four Promise Combinators

👥 Promise.all ALL must succeed. One fails = all fail. Group dinner: everyone must arrive.
📊 Promise.allSettled Collect ALL results (success or fail). Survey: gather every response.
🏅 Promise.race FIRST to settle wins. Taxi vs Uber: whichever arrives first.
🎯 Promise.any FIRST to succeed. Only fails if ALL fail. Backup servers.

async/await — Same Promise, Better Syntax

Promise Chain


fetchUser(1)
  .then(user =>
    fetchOrders(user.id))
  .then(orders =>
    console.log(orders))
  .catch(err =>
    console.error(err));
=

async/await


async function load() {
  try {
    const user = await fetchUser(1);
    const orders = await fetchOrders(user.id);
    console.log(orders);
  } catch (err) {
    console.error(err);
  }
}
⚡ try/catch with await is the SAME try/catch from Lecture 13 — now applied to async code!

Let's see this in code!

Promise.all, allSettled, race, any + async/await


// Parallel with async/await + Promise.all
async function loadAllCities() {
  const cities = await Promise.all([
    getWeather("Lahore"),
    getWeather("Karachi"),
    getWeather("Islamabad")
  ]);
  cities.forEach(c => console.log(c.city, c.temp));
}

Part 3 — Predict Output

What happens with Promise.all?


const results = await Promise.all([
  Promise.resolve("A"),
  Promise.reject(new Error("B failed")),
  Promise.resolve("C")
]);
console.log(results);
  1. ["A", Error, "C"]
  2. ["A", undefined, "C"]
  3. console.log never runs — Promise.all rejects immediately ✓
  4. ["A", "C"] (skips the failed one)

One rejection cancels everything. Use Promise.allSettled for partial results.

Part 3 — Concept Challenge

When is allSettled better than all?

  1. When you need the fastest result only
  2. Dashboard loading — show successful data even if some sources fail ✓
  3. When all requests must succeed for a transaction
  4. When you need to cancel other requests after the first succeeds

allSettled = dashboards. all = transactions. Different use cases.

Part 3 — Spot the Error

What's wrong with this code?


async function getUser() {
  const response = fetch("/api/user");
  const data = response.json();
  return data;
}
const user = getUser();
console.log(user.name);
  1. Missing await — response and data are Promise objects, not values ✓
  2. fetch doesn't work inside async functions
  3. Should use .then() instead of const assignment
  4. getUser() needs parentheses with arguments

Part 3 Summary

  • Common Mistake: Sequential await for independent requests — use Promise.all for parallel execution
  • Best Practice: Combine async/await with Promise.all: const [a, b] = await Promise.all([...])
  • Pro Tip: Refactoring sequential await to Promise.all can cut page load from 8s to 2s — the #1 async optimization.

Part 4

The fetch API & REST APIs

Making HTTP requests from the browser

fetch() — Two Failure Points

💻 Your Code fetch(url)
→
❌ Network Error REJECTS (catch)
→
⚠ HTTP 404/500 RESOLVES! Check .ok
→
✅ HTTP 200 Success!
❌ Critical: fetch does NOT reject on 404/500! You MUST check response.ok yourself. This is the #1 fetch mistake.

Loading / Error / Success — The Frontend Trinity

🔄 IDLE
click →
⏳ LOADING Show spinner
→
✅ SUCCESS Show data
↓ error ↓
❌ ERROR Show message
⚡ This is exactly how React useState works: isLoading, data, error

Let's see this in code!

fetch + response.ok + AbortController + parallel loading


async function fetchData(url) {
  try {
    const response = await fetch(url);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error(`Fetch failed: ${error.message}`);
    return null;
  }
}

Part 4 — Spot the Error

What's the subtle bug?


async function getUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    const user = await response.json();
    return user;
  } catch (error) {
    console.error("Failed:", error);
  }
}
  1. Missing return in catch block
  2. No response.ok check — 404/500 responses are parsed as success ✓
  3. Should use .then() instead of await
  4. The catch block should re-throw the error

Part 4 — Concept Challenge

Why does fetch need TWO error checks?

  1. Because JavaScript has two types of exceptions
  2. Because the browser runs two separate processes
  3. Network errors and HTTP errors are different layers — a 404 IS a successful network response ✓
  4. It's a design flaw in the fetch specification

Like mail: the postal service delivered the response (network OK), but the answer says "not found" (HTTP 404).

Part 4 Summary

  • Common Mistake: Not checking response.ok — fetch resolves on 404 and 500
  • Best Practice: Build a reusable fetchData helper with error handling, response.ok check, and timeout
  • Pro Tip: Use AbortController to cancel requests — essential for React useEffect cleanup

Sequential vs Parallel Execution

Sequential (Slow)


// Each waits for the previous
const a = await fetch(url1); // 1s
const b = await fetch(url2); // 1s
const c = await fetch(url3); // 1s
// Total: ~3 seconds
VS

Parallel (Fast!)


// All start at the same time
const [x, y, z] = await Promise.all([
  fetch(url1),
  fetch(url2),
  fetch(url3)
]);
// Total: ~1 second

Let's see this in code!

Real weather data from Open-Meteo API + retry logic + TypeScript types


// Reusable typed fetch helper
async function fetchJSON(url) {
  const response = await fetch(url, {
    signal: AbortSignal.timeout(10000)
  });
  if (!response.ok) throw new Error(`HTTP ${response.status}`);
  return response.json();
}

  2. A: ~1s, B: ~3s

Part 5 — Concept Challenge

What does Promise<WeatherData> tell TypeScript?

  1. The function is synchronous and returns WeatherData
  2. The function returns a Promise that resolves to a WeatherData object ✓
  3. WeatherData is a Promise subclass
  4. The function might return WeatherData or null

You get full autocomplete: .city, .temperature, .windSpeed — with type checking!

Part 5 Summary

  • Common Mistake: Sequential await for independent requests — always ask "do these depend on each other?"
  • Best Practice: Build a reusable typed fetchJSON<T> helper — one function, consistent error handling everywhere
  • Pro Tip: Implement retry with exponential backoff for production resilience: 2s, 4s, 8s delays

The fetch API Pattern


async function fetchData(url) {
  const response = await fetch(url);
  if (!response.ok) {                  // Check HTTP errors!
    throw new Error(`HTTP ${response.status}`);
  }
  return await response.json();        // Also async!
}

Remember: fetch does NOT reject on 404/500 — check response.ok!

AbortController


// Modern timeout
const response = await fetch(url, {
  signal: AbortSignal.timeout(5000)
});

// Manual cancel
const controller = new AbortController();
fetch(url, { signal: controller.signal });
controller.abort();  // Cancel the request
