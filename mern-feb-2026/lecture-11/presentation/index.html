Part 1

Functions — Your First Reusable Building Blocks

A function is a named, reusable block of code that accepts input, performs a task, and returns output.

💡 Analogy: A function is like a vending machine — you insert money (input), it processes your request (body), and gives you a snack (return value).

Function Anatomy


function calculateBonus(salary, rating) {
    const bonus = salary * (rating / 10);
    return bonus;
}

const result = calculateBonus(50000, 8);  // 40000
Keywords Function Name Parameters Function Body

Three Ways to Write Functions

What does greet("Sara") return?

    `Hello, ${name}! Welcome aboard.`

  2. undefined ✓
  3. ReferenceError

Curly braces = explicit return required. Missing return means undefined.

Part 1 Summary

  • Common Mistake: Returning an object without ( ) — () => ({ key: val })
  • Best Practice: Use const for all function expressions and arrow functions
  • Pro Tip: Name functions descriptively: calculateAnnualSalary not calc
💎 25-year insight: The most maintainable codebases have small, well-named functions. Code that reads like English.

Part 2

Scope, Closures & Pure Functions

Scope determines where variables live. Closures are functions that remember their enclosing scope.

The Scope Chain

Global Scope
Function Scope (outer)
Function Scope (inner)
Block Scope (if/for)

Lookup goes upward only — inner can see outer, never the reverse.

Three Levels of Scope

ScopeCreated ByVisibility
GlobalTop-level codeEverywhere
Functionfunction keywordInside that function
Block{ } with let/constInside that block
⚠ var ignores block scope — another reason we never use var.

Closures — Functions That Remember


function createCounter(startValue = 0) {
    let count = startValue;  // ← "lives on" in the closure

    return {
        increment: () => ++count,
        decrement: () => --count,
        getCount: () => count
    };
}

const counter = createCounter();
counter.increment();  // 1
counter.increment();  // 2
counter.getCount();   // 2  ← count persists!
⚡ Each call to createCounter() creates a new, independent closure with its own count.

Part 2 Summary

  • Common Mistake: Assuming closures capture values (they capture references)
  • Best Practice: Use closures for data privacy — expose only what's needed
  • Pro Tip: Pure functions = same input, same output, no side effects. React components should be pure.
💎 25-year insight: React hooks — useState, useEffect, useCallback — are all closures. Understanding closures = understanding React.

Part 3

Callbacks & Higher-Order Functions

forEach, map, filter, find — the modern way to process arrays.

From for Loop to HOF

Imperative (Old)


const results = [];
for (let i = 0; i < employees.length; i++) {
    results.push(
        employees[i].name.toUpperCase()
    );
}
VS

Declarative (Modern)


const results = employees.map(
    emp => emp.name.toUpperCase()
);

Same result. Half the code. Zero indexing bugs.

How map Works

📜 "sara"
→
🔨 toUpperCase
→
✅ "SARA"

Each element goes through the callback. A new array is returned. Original untouched.

⚡ forEach = side effects (logging). map = transformation (new array). filter = selection (subset).

Part 3 — Interactive Question

What does this return?


const nums = [1, 2, 3, 4, 5];
const result = nums.map(n => n * 2).filter(n => n > 6);
console.log(result);
  1. [6, 8, 10]
  2. [7, 8, 9, 10]
  3. [8, 10] ✓
  4. [4, 5]

map doubles: [2,4,6,8,10] → filter keeps > 6: [8,10]. This is method chaining!

Part 3 — Quick-Fire Recall

Name the Method!

Get only employees with salary > 70000filter
Get every employee's name as an arraymap
Find the first employee in Salesfind
Log each employee to the consoleforEach
Get the index of first inactive employeefindIndex

Part 3 Summary

  • Common Mistake: Using forEach when you need a result — use map instead
  • Best Practice: Keep callbacks pure. Use find instead of filter()[0]
  • Pro Tip: In React, {users.map(u => <Card />)} is how you render lists. These are your daily tools.

Part 4

some, every, reduce, sort & Chaining

Complete the HOF toolkit — then chain them into data pipelines.

The Complete HOF Toolkit

MethodReturnsPurpose
forEachundefinedIterate (side effects)
mapNew array (same length)Transform each element
filterNew array (≤ length)Select by condition
findElement or undefinedFirst match
some / everybooleanAny / all pass?
reduceAny valueAccumulate to single result
sortSame array (mutated!)Order elements

How reduce Works


[10, 20, 30].reduce((sum, num) => sum + num, 0)
sum=0, num=10 → 10
→
sum=10, num=20 → 30
→
sum=30, num=30 → 60

The accumulator carries the running result through each element.

Part 4 — Interactive Question

What gets logged?


const scores = [72, 85, 91, 68, 95, 88];
const allPassing = scores.every(s => s >= 70);
const anyPerfect = scores.some(s => s === 100);
console.log(allPassing, anyPerfect);
  1. true, false
  2. true, true
  3. false, false ✓
  4. false, true

68 fails the ≥70 test, and no score is 100. every short-circuits at first failure.

Part 4 — Spot the Error

What's wrong with this code?


const prices = [200, 30, 1000, 5, 80];
console.log(prices.sort());
// Expected: [5, 30, 80, 200, 1000]
  1. Nothing — it works correctly
  2. Default sort is lexicographic — "1000" < "200" ✓
  3. sort() doesn't work on numbers
  4. prices is const, can't be sorted

Fix: [...prices].sort((a, b) => a - b) — always use a comparator for numbers!

Part 4 Summary

  • Common Mistake: sort() mutates the original! Always sort a copy: [...arr].sort()
  • Best Practice: Filter first (reduce data), then sort, then map. Chain in logical order.
  • Pro Tip: Use reduce for totals and grouping. For everything else, map/filter are more readable.

Part 5

TypeScript Functions

Parameter types, return types, optional params & generics preview.

TypeScript Function Types


// Parameter types + return type
function calculateBonus(salary: number, rating: number): number {
    return salary * (rating / 10);
}

// Arrow function with types
const formatEmployee = (name: string, dept: string): string => {
    return `${name} (${dept})`;
};

// Optional parameter
const createProfile = (
    name: string,
    role: string = "Junior Developer",
    department?: string
): { name: string; role: string; department: string } => {
    return { name, role, department: department ?? "Unassigned" };
};

Generics — Functions for Any Type


// T is a type variable — filled in when you call the function
function getFirst<T>(items: T[]): T | undefined {
    return items[0];
}

const firstNum = getFirst([10, 20, 30]);       // number
const firstStr = getFirst(["hello", "world"]); // string
💎 React preview: useState<number>(0) is a generic! Understanding generics now makes Module 3 smoother.

Part 5 Summary

  • Common Mistake: Using any to bypass type errors — defeats TypeScript's purpose
  • Best Practice: Always type function parameters. Return types can often be inferred.
  • Pro Tip: Start with basic <T> generics. You'll use them daily in React: useState<string>, useRef<HTMLInputElement>

Cheat Sheet: Chaining Pattern


// The data pipeline pattern
data
  .filter(condition)     // narrow down
  .sort(comparator)      // order
  .map(transform)        // reshape
  .forEach(action)       // output
⚡ Remember: Filter first (reduce data), then sort (smaller dataset), then map (transform), then forEach (output).
