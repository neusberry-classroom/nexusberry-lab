Part 1

String Fundamentals & Core Methods

User names, emails, search queries, API responses, URLs — all strings.
To work with real-world data, you need string methods.

💡 Key Concept: Strings are immutable. Every method returns a NEW string — the original is never modified. Think of strings like printed text on paper — you can't erase a letter, you print a new page.

String Methods Categories

🔍
Search
indexOf, lastIndexOf, includes, startsWith, endsWith
🔄
Transform
toUpperCase, toLowerCase, trim, replace, replaceAll, padStart, padEnd
✂️
Extract
slice, charAt, at, split
⚠️ Every method returns a new value — the original string is never modified. You must capture the return: const cleaned = input.trim()

Character Access


const title = "Smart Content Analyzer";

title.length;          // 22
title.charAt(0);       // "S"
title[6];              // "C"  (bracket notation)
title.at(-1);          // "r"  (negative = from end)
title[title.length-1]; // "r"  (old way)
💎 at(-1) is cleaner than str[str.length - 1]. Added in ES2022 — works on strings and arrays.

Searching Strings


const article = "JavaScript is versatile. JavaScript powers the web.";

article.indexOf("JavaScript");       // 0
article.lastIndexOf("JavaScript");   // 24
article.indexOf("Python");           // -1 (not found)
article.includes("versatile");       // true
article.startsWith("JavaScript");    // true
article.endsWith("web.");            // true
MethodReturnsUse When
indexOf()Position / -1Need the location
includes()BooleanJust need yes/no
startsWith()BooleanChecking prefix
endsWith()BooleanChecking suffix

Extract, Transform & Replace


// Extracting
"https://www.nexusberry.com".slice(8);   // "www.nexusberry.com"
"Hello World".slice(-5);                  // "World"

// Transforming
"   John DOE   ".trim();                 // "John DOE"
"Hello".toLowerCase();                    // "hello"
"Hello".toUpperCase();                    // "HELLO"

// Replacing
"I love cats".replace("cats", "dogs");      // first only!
"cats cats".replaceAll("cats", "dogs");     // all matches
⚠️ replace() only replaces the first occurrence! Use replaceAll() for all matches, or regex with /g flag.

Split, Join & Template Literals


// Split — string to array
"Laptop,Mouse,Keyboard".split(",");  // ["Laptop","Mouse","Keyboard"]

// Join — array to string
["A", "B", "C"].join(" | ");         // "A | B | C"

// Word count
"Hello world today".split(/\s+/).length;  // 3

// Template literals — multiline + expressions
const product = { name: "Laptop", price: 89999 };
console.log(`${product.name}: Rs. ${product.price.toLocaleString()}`);

// Padding & Repeating
"5".padStart(3, "0");     // "005"
"Hi".padEnd(10, ".");     // "Hi........"
"─".repeat(30);           // 30 dashes

🔎 Spot the Error


const input = "  Hello  ";
input.trim();
if (input === "Hello") {
    console.log("Match!");
}
  • The if condition should use ==
  • trim() returns a new string — result not captured
  • trim() doesn't exist on strings
  • The spaces are already removed
💥 #1 string bug: Strings are immutable! input.trim() returns a new string. Fix: const cleaned = input.trim()

Part 1 Summary

⚠️ Common Mistake: Calling string methods without capturing the return value. str.trim() doesn't modify str!
✅ Best Practice: Always trim() user input and toLowerCase() before comparisons.
💎 Pro Tip: Use slice() over substring() — it supports negative indices and is the modern standard.

Part 2

Regular Expressions

Pattern Matching Fundamentals

String methods handle exact matches. Regex handles patterns.
"Find any phone number." "Validate email format." "Replace all dates."

💡 Analogy: Regex is like a combination lock — each slot accepts certain characters. The pattern defines the shape, not the exact value.

Regex Building Blocks

🔡
Character Classes
\d (digit), \w (word), \s (space), . (any)
🎯
Character Sets
[a-z], [0-9], [aeiou], [^0-9] (negation)
🔢
Quantifiers
* (0+), + (1+), ? (0-1), {n}, {n,m}
⚓
Anchors
^ (start), $ (end), \b (word boundary)
🔗
Groups
() capture, | alternation (OR)

Anatomy of a Regex


  /^\d{3}-\d{3}-\d{4}$/g
   │ │  │  │  │  │   │ │
   │ │  │  │  │  │   │ └─ flag: g = global
   │ │  │  │  │  │   └─── $ = must end here
   │ │  │  │  │  └─────── {4} = exactly 4 digits
   │ │  │  │  └────────── \d = any digit
   │ │  │  └───────────── - = literal dash
   │ │  └──────────────── {3} = exactly 3 digits
   │ └─────────────────── \d = any digit
   └───────────────────── ^ = must start here

Character Classes


"Order #42 on 2024-01-15".match(/\d+/g);
// ["42", "2024", "01", "15"]

"hello world!".match(/\w+/g);
// ["hello", "world"]

// Uppercase = negation
// \d → digit     \D → NON-digit
// \w → word char \W → NON-word char
// \s → space     \S → NON-space
ClassMatchesEquivalent
\dAny digit[0-9]
\wWord character[a-zA-Z0-9_]
\sWhitespace[ \t\n\r]
.Any char (not newline)—

Quantifiers & Anchors


// Quantifiers
/a+/.test("aaa");     // true — 1 or more
/colou?r/.test("color");  // true — u is optional
/\d{3}/.test("123");  // true — exactly 3 digits

// Anchors
const text = "cat catalog scattered";
text.match(/cat/g);       // ["cat","cat","cat"] — everywhere
text.match(/^cat/g);      // ["cat"] — only at start
text.match(/\bcat\b/g);   // ["cat"] — whole word only
⚠️ \b = word boundary. Without it, /cat/ matches inside "catalog" and "scattered". With \bcat\b, only the standalone word matches.

Groups, Flags & Methods


// Capturing groups — extract parts
const dateMatch = "2024-01-15".match(/(\d{4})-(\d{2})-(\d{2})/);
// dateMatch[1] = "2024", [2] = "01", [3] = "15"

// Flags: g = global, i = case-insensitive
"Hello hello HELLO".match(/hello/gi);  // ["Hello","hello","HELLO"]

// Methods
/\d/.test("abc123");         // true (boolean — fastest)
"abc123".search(/\d/);       // 3 (index)
"a1b2c3".match(/\d/g);      // ["1","2","3"] (all matches)
"cats CATS".replace(/cats/gi, "dogs"); // "dogs dogs"

💡 Predict the Output


const result = "abc 123 def 456".match(/\d+/g);
console.log(result);

💡 \d+ matches sequences of digits (1 or more). The g flag finds ALL matches.

🔎 Spot the Error

This should validate that a string is ONLY digits:


const isAllDigits = /\d+/.test("abc123");
console.log(isAllDigits); // true — but "abc123" isn't all digits!
  • Should use \D instead of \d
  • Should use match() instead of test()
  • Missing anchors — needs /^\d+$/
  • Needs the g flag
💥 Always anchor validation: /\d+/ = "contains digits somewhere." /^\d+$/ = "is ENTIRELY digits."

Part 2 Summary

⚠️ Common Mistake: Forgetting ^ and $ anchors in validation regex — accepts partial matches.
✅ Best Practice: Escape special characters with \: \. for literal dot, \/ for literal slash.
💎 Pro Tip: Use test() for validation (boolean), match() for extraction. Prefer /pattern/ over new RegExp() unless the pattern is dynamic.

Practical Regex Patterns

Real-World Validation

Email, phone, password, URL — the four most common validation tasks in web development.

The Validation Layer Cake

🎨 Layer 1: HTML5 Attributes type="email", required, pattern
↓
🔎 Layer 2: JavaScript Regex ← TODAY Real-time format validation
↓
🛠️ Layer 3: Business Logic Password strength, username rules
↓
🔒 Layer 4: Server Validation Never trust client-side alone

Email & Phone Validation


// Email — practical format check
const emailRegex = /^[\w.-]+@[\w.-]+\.\w{2,}$/;
emailRegex.test("user@example.com");     // true
emailRegex.test("invalid@");             // false

// Pakistan phone — multiple formats
const phoneRegex = /^(\+92|0)\d{3}-?\d{7}$/;
phoneRegex.test("0300-1234567");         // true
phoneRegex.test("+923001234567");        // true
phoneRegex.test("300-1234567");          // false (missing prefix)
⚠️ Regex validates format, not existence. Send a confirmation email for real validation.

Password Strength & URL Validation


// Password — separate checks (better UX than one mega-regex)
password.length >= 8;           // Length
/[A-Z]/.test(password);        // Uppercase
/[a-z]/.test(password);        // Lowercase
/\d/.test(password);           // Digit
/[!@#$%^&*]/.test(password);  // Special character

// URL validation
const urlRegex = /^https?:\/\/[\w.-]+(:\d+)?(\/[\w./\-?&#=]*)?$/;
urlRegex.test("https://www.nexusberry.com");  // true
urlRegex.test("ftp://files.example.com");     // false
💎 Multiple simple patterns > one massive regex. Granular feedback: "Missing uppercase" is better than "Invalid password."

Data Extraction from Text


const text = `
  Email: admin@nexusberry.com
  Phone: 0328-4500073
  Site: https://www.nexusberry.com
`;

text.match(/[\w.-]+@[\w.-]+\.\w{2,}/g);  // ["admin@nexusberry.com"]
text.match(/\d{3,4}-\d{7,8}/g);           // ["0328-4500073"]
text.match(/https?:\/\/[\w.-]+/g);         // ["https://www.nexusberry.com"]
⚠️ match() returns null when no matches found — NOT an empty array! Guard with || []

💡 Hidden Fact Reveal

What does match() return when there are NO matches?


"abc".match(/\d/g)
  • []
  • undefined
  • null
  • false
💥 null, not []! Calling .length or .join() on null crashes. Always guard: str.match(/pattern/g) || []

Part 3 Summary

⚠️ Common Mistake: Unescaped . matches ANY character. Use \. for a literal dot in URLs and emails.
✅ Best Practice: Client-side regex is for UX. Server-side validation is for security. Never trust client-side alone.
💎 Pro Tip: Break complex validation into multiple simple patterns. Test against both valid AND invalid input.

Part 4

Defensive Error Handling

try / catch / finally

Things go wrong. Users enter garbage. APIs go down. Data arrives in unexpected formats.
Error handling is how professional code survives the unexpected.

💡 Analogy: try/catch is a safety net at a circus. The trapeze artist (try) attempts the trick. If they fall, the net catches them (catch). The crew always cleans up (finally).

JavaScript Error Types

🔴
TypeError
null.toString() — wrong type operation. Most common in production!
🔵
ReferenceError
console.log(x) — using an undeclared variable
🟠
RangeError
new Array(-1) — value out of allowed range
🟢
SyntaxError
if (true { — invalid code structure (caught at parse time)

try / catch / finally Flow


try {
    line 1  ✅ runs
    line 2  ✅ runs
    line 3  💥 ERROR — jumps to catch
    line 4  ⏭️ SKIPPED (never runs)
} catch (error) {
    handle error ✅ runs
} finally {
    cleanup ✅ ALWAYS runs
}

next code ✅ continues (program NOT crashed)
💡 Without try/catch, the error crashes the program. With it, the error is handled and execution continues.

try/catch in Practice


function parseJSON(jsonString) {
    let result = null;
    try {
        result = JSON.parse(jsonString);
        console.log("✅ Parsed:", result);
    } catch (error) {
        console.log(`❌ Failed: ${error.message}`);
        result = null;
    } finally {
        console.log("🔄 Cleanup (always runs)");
    }
    return result;
}

parseJSON('{"name": "Laptop"}'); // ✅ Works
parseJSON('invalid');             // ❌ Caught, continues

throw — Custom Errors


function validateAge(age) {
    if (typeof age !== "number") {
        throw new TypeError(`Age must be a number, got ${typeof age}`);
    }
    if (age < 0 || age > 150) {
        throw new RangeError(`Age must be 0-150, got ${age}`);
    }
    return `✅ Valid age: ${age}`;
}

try {
    validateAge("thirty");
} catch (error) {
    console.log(`${error.name}: ${error.message}`);
    // TypeError: Age must be a number, got string
}

Guard Clauses — Fail Fast

❌ Nested If/Else


function process(user) {
  if (user) {
    if (user.name) {
      if (user.name.length > 0) {
        return `OK: ${user.name}`;
      }
    }
  }
  return "Invalid";
}
vs

✅ Guard Clauses


function process(user) {
  if (!user) return "No user";
  if (!user.name) return "No name";
  if (!user.name.trim()) return "Empty";

  return `OK: ${user.name}`;
}
💎 Guard clauses: validate at the top, return early on failure. Happy path stays at lowest indentation.

Combining validation, regex & error handling

⚠️ Critical: Today's try/catch is the foundation for every async/await call in Lecture 14. Every fetch() needs error handling.

💡 Predict the Output


try {
    console.log("A");
    throw new Error("oops");
    console.log("B");
} catch (error) {
    console.log("C");
} finally {
    console.log("D");
}
console.log("E");
  • A, B, C, D, E
  • A, C, D
  • A, C, D, E
  • A, C, E
💡 "B" is skipped (after throw). "D" always runs (finally). "E" runs because error was caught.

💡 Concept Challenge

Should you wrap EVERY line of code in try/catch?

  • Yes — always be safe
  • Only in production code
  • No — only wrap code that can fail unpredictably
  • Only for async code
💎 Over-catching hides bugs. Wrap boundaries: user input, JSON parsing, API calls. Let unexpected bugs crash loudly so you can fix them.

Part 4 Summary

⚠️ Common Mistake: Empty catch blocks — silently swallowing errors. Always log or return a status.
✅ Best Practice: Guard clauses for expected conditions. try/catch for unpredictable failures. Never catch and ignore.
💎 Pro Tip: The most expensive production bugs aren't crashes — they're silent failures. Code that catches errors, returns defaults, and produces wrong results.

// String union types — restrict values
type Status = "loading" | "success" | "error";

// Template literal types — generate combinations

// Exhaustive switch
function handle(status: Status) {
    switch (status) {
        case "loading": return "⏳";
        case "success": return "✅";
        case "error":   return "❌";
    } // TypeScript ensures ALL cases covered
}
💎 Template literal types will be used extensively in React for component prop types: "small" | "medium" | "large"

💡 Concept Challenge

Why use guard clauses instead of wrapping everything in try/catch?

  • Guard clauses are faster to type
  • Guards handle expected conditions; try/catch handles unexpected failures
  • try/catch doesn't work with strings
  • Guard clauses are newer JavaScript
💡 Guards are proactive (prevent). try/catch is reactive (recover). Using exceptions for flow control is an anti-pattern.

Part 5 Summary

⚠️ Common Mistake: Using split(" ") for word count — double spaces create empty strings. Use split(/\s+/) instead.
✅ Best Practice: Return structured objects { success, data, error } from functions that can fail.
💎 Pro Tip: The content analyzer pattern — validate, process, extract, report — is used by every CMS, email platform, and document editor.
